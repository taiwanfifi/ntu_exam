# 寫作技法總結 — 行動指南

> 從 25 部參考作品中提煉出的 8 個核心寫作模式，
> 以及《演算法偵探筆記》接下來可以具體採用的策略。

---

## 一、8 大核心寫作模式

### 模式 1：對話教學法（數學少女模式）
```
來源：《數學少女》
核心：角色對話中自然嵌入概念推導

做法：
  ✓ 「如果我們不從第一行開始，而是直接看中間呢？」
  ✗ 「Binary Search 的定義是在已排序陣列中......」

我們的狀態：✅ 已經在用，且用得很好。繼續。
```

### 模式 2：期限驚悚法（Phoenix Project 模式）
```
來源：The Phoenix Project
核心：用 deadline 壓力讓技術學習充滿緊迫感

做法：
  - 「天亮前必須恢復伺服器」
  - 「入侵者可能還在系統裡」
  - 「考試下週三，但你還不會 DP」

我們的狀態：✅ 第一夜已建立（凌晨案件）。後續章節繼續提高stakes。
```

### 模式 3：日常之謎法（冰菓模式）
```
來源：《冰菓》
核心：演算法不需要大場面，生活問題就夠了

做法：
  - 「食堂排隊為什麼今天特別慢？」→ Queue 的概念
  - 「為什麼搜尋引擎比你翻資料快？」→ Indexing
  - 「為什麼你的 WiFi 密碼猜不到？」→ Hash function

我們的狀態：⬜ 可以在「日常篇」中引入，作為緊張章節之間的喘息。
```

### 模式 4：關卡遞增法（死亡萬花筒模式）
```
來源：《死亡萬花筒》
核心：每個章節弧線教一個新概念，難度遞增

做法：
  門 1（Easy）：Linear Search / Two Sum
  門 2（Easy+）：Binary Search
  門 3（Medium）：Hash Map + DFS
  ...
  門 12（Hard）：DP + Graph Theory

我們的狀態：⬜ 可做為整體結構的骨架。每個「案件」= 一扇門。
```

### 模式 5：程式碼嵌入法（Cryptonomicon 模式）
```
來源：Cryptonomicon
核心：直接在敘事中嵌入可運行的程式碼

做法：
  學姊在鍵盤上打出：

  seen = set()
  duplicates = []
  for record in data:
      if record.id in seen:
          duplicates.append(record)
      else:
          seen.add(record.id)

  「跑完了。2,627 筆重複。」

我們的狀態：⚠️ 目前用 pseudo code，可以進一步用真正的 Python。
```

### 模式 6：超短章節法（Dan Brown 模式）
```
來源：Digital Fortress
核心：2-4 頁的極短章節，讓讀者停不下來

做法：
  章節結構：
  ├── 場景設定（2頁）
  ├── 問題提出（1頁）
  ├── 推理過程（3-5頁）
  ├── 揭示（2頁）
  └── 鉤子（最後一句留懸念）

我們的狀態：⚠️ 第一夜的每個「幕」偏長。可以拆得更碎。
```

### 模式 7：硬核引擎法（三體模式）
```
來源：《三體》
核心：科學嚴謹性本身就是劇情引擎

做法：
  不是「演算法可以解決問題」→ 弱
  而是「如果不用正確的演算法，一切都會崩潰」→ 強

  場景：資料庫有 100 萬筆，暴力解要 83 分鐘。
        但入侵者隨時可能再次行動。
        你只有 2 秒的時間。
        你 必 須 用 O(n)。

我們的狀態：✅ 已經做得很好（「五千億次 vs 兩秒」的對比）。繼續。
```

### 模式 8：多格式混搭法（Listen for the Lie 模式）
```
來源：Listen for the Lie
核心：散文+程式碼區塊+debug日誌+問題描述交替

做法：
  混搭這些格式——
  ├── 第一人稱敘事
  ├── Terminal 輸出
  ├── 偵探手札（知識框）
  ├── 暫停推理（互動區）
  ├── ASCII 圖表
  └── Debug 日誌

我們的狀態：✅ 已經有偵探手札、暫停推理、ASCII 圖。非常好。
```

## 二、《演算法偵探筆記》的當前優勢

根據第一夜的內容分析，我們已經自然運用了以下技法：

| 技法 | 使用狀態 | 具體例子 |
|------|----------|----------|
| 對話教學法 | ✅ 優秀 | 林凱和阿翔的 Binary Search 對話 |
| 期限驚悚法 | ✅ 良好 | 凌晨三點的緊急案件 |
| 程式碼嵌入 | ⚠️ 半用 | pseudo code 有，真正 Python 可以更多 |
| 多格式混搭 | ✅ 優秀 | 偵探手札 + 暫停推理 + ASCII art |
| 硬核引擎 | ✅ 優秀 | O(n²) vs O(n) 的戲劇化呈現 |
| 三角角色 | ✅ 良好 | 學姊(邏輯) + 林凱(直覺) + 阿翔(暴力) |
| 鉤子密度 | ✅ 良好 | 幾乎每 2-3 頁有一個轉折 |
| 幽默感 | ✅ 良好 | 阿翔的笑話、學姊的冷回應 |

## 三、後續章節的行動清單

### 結構層面
- [ ] 規劃 15-20 章的整體結構（每章 = 一個核心演算法/資料結構）
- [ ] 建立「關卡遞增」的難度曲線（Easy → Medium → Hard）
- [ ] 設計「長線懸念」：資料庫裡的隱藏圖網路最終指向什麼？
- [ ] 加入「日常篇」作為緊張章節之間的喘息（冰菓模式）

### 角色層面
- [ ] 深化學姊的「O(1) 性格」：她為什麼這麼厲害？有沒有弱點？
- [ ] 給阿翔「暴力美學」的高光時刻：有些場景，O(n²) 反而是對的
- [ ] 林凱的成長弧線：從「靠直覺」→「直覺+邏輯的結合」
- [ ] 入侵者的身份和動機：讓讀者也在猜
- [ ] 考慮引入新角色（面試官？業界工程師？另一個競爭團隊？）

### 技法層面
- [ ] 引入更多真正的 Python 程式碼（讀者可以實際執行）
- [ ] 嘗試「入侵者視角」的穿插章節（貓鼠遊戲模式）
- [ ] 加入「三年後的面試場景」作為預示（非線性時間）
- [ ] 考慮「超短章節」拆分，讓單個章節不超過 10 頁

### 教學層面
- [ ] 每個演算法都要有「偵探手札」做知識總結
- [ ] 每個演算法都要有「暫停推理」讓讀者先想
- [ ] 增加「為什麼不能用暴力解」的戲劇化場景
- [ ] 加入「常見面試問題」的暗示，讓讀者感覺「學這個真的有用」

## 四、推薦優先閱讀的 5 部作品

如果只能讀 5 部作為參考，按優先度排序：

1. **《數學少女》**：結構範本。看它怎麼在對話中教數學。
2. **《冰菓》**：語調範本。看它怎麼用日常謎團教邏輯推理。
3. **《詭秘之主》**（前 50 章即可）：世界觀範本。看它怎麼建構系統化知識體系。
4. **《三體》第一部**：「硬核概念驅動劇情」的最佳示範。
5. **《約定的夢幻島》第一弧**（前 37 章）：懸疑節奏的天花板。

## 五、最終提醒

```
┌─────────────────────────────────────────────────┐
│                                                   │
│  記住：讀者是來看故事的，不是來上課的。            │
│                                                   │
│  最好的教學發生在讀者意識不到自己在學習的時候。    │
│                                                   │
│  如果一段話刪掉也不影響故事，那它可能不該存在。    │
│  如果一段知識不透過故事傳遞，那它可能不該出現。    │
│                                                   │
│  永遠問自己：                                     │
│  「如果讀者對演算法完全不感興趣，                  │
│    他還會不會想繼續讀下去？」                      │
│                                                   │
│  如果答案是 yes，你就成功了。                      │
│                                                   │
```
